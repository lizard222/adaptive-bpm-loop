# REFLEX-BPM

**Адаптивное управление бизнес-процессами вуза: система исполняет регламентированные процессы, ведёт журнал событий, сама находит в нём расхождения с регламентом и корректирует параметры процессов на следующий цикл.**

> Прототип к магистерской диссертации «Адаптивное управление бизнес-процессами вуза на основе интеллектуального анализа журналов событий и многоагентного управления исполнением процессов» (ВолгГТУ, кафедра САПРиПК).

---

## Идея

Нормативный регламент («как должно быть») и реальное исполнение процесса («как получилось») всегда расходятся: этапы просрочиваются, шаги пропускаются, узкие места повторяются из семестра в семестр. Обычно это замечают вручную и постфактум.

REFLEX-BPM замыкает контур — как рефлекторная дуга:

```
исполнение BPMN-процессов ──► журнал событий (XES)
        ▲                            │
        │                            ▼
корректировки параметров ◄── интеллектуальный анализ (pm4py):
(сдвиг сроков, новые         discovery + conformance checking —
 контрольные точки)          где и насколько факт разошёлся с регламентом
```

Найденные систематические отклонения автоматически превращаются в предложения по корректировке (режим «система советует — человек решает»), и следующий цикл процесс идёт уже с уточнёнными параметрами. Главный научный результат работы — алгоритм преобразования расхождений в корректировки и экспериментальное подтверждение того, что метрики (доля просрочек, среднее опоздание, число эскалаций) улучшаются от цикла к циклу.

## Пилотные процессы

Система намеренно ограничена **двумя** регламентированными процессами кафедры (по реальным нормативным документам):

| Процесс | Почему выбран |
|---|---|
| **Организация защиты ВКР** | жёсткие нормативные дедлайны («приказ о допуске — за 14 дней до ГЭК»), много участников, эскалации |
| **Планирование учебной нагрузки** | цикличность 2 раза в год — проверка переносимости контура на другой тип процесса |

## Возможности

- 🗂 **Каталог процессов** — исполняемые модели BPMN 2.0 с метаданными: роли, сроки, шаблоны документов, правила запуска
- 🚀 **Автозапуск** — Планировщик стартует экземпляры по академическому календарю (его параметры — объект адаптации)
- ⚙️ **Исполнение** — на каждый экземпляр порождается агент процесса, ведущий BPMN как конечный автомат
- ✅ **Задачи для людей** — личный кабинет: список задач, сроки, структурированные действия («Утвердить» / «Отклонить» с причиной)
- ⏰ **Контроль сроков** — Контролер: напоминания и лестница эскалации (исполнитель → руководитель → уровень выше)
- 📄 **Документы из шаблонов** — проекты приказов и служебных записок (.docx) с автоподстановкой данных экземпляра
- 📊 **Журнал событий** — каждое действие: `case_id, activity, timestamp, resource`; экспорт в XES
- 🔁 **Контур адаптации** — анализ журнала за цикл → классификация расхождений → корректировки параметров (`propose`)
- 🧪 **Имитационный режим** — прогон N «семестров» на синтетических исполнителях с фиксируемым seed — среда воспроизводимого эксперимента

## Архитектура

```
Веб-интерфейс (React: кабинет задач, панель расхождений и корректировок)
        │
API-шлюз (FastAPI: REST + WebSocket; JWT-авторизация по ролям)
        │
Оркестратор процессов (SpiffWorkflow: исполнение BPMN 2.0;
                       состояние экземпляров — в PostgreSQL)
        │
┌───────────────────────────────┐    ┌────────────────────────────────┐
│ Слой агентов (SPADE, FIPA ACL)│◄──►│ Сервис анализа (pm4py)         │
│ • Планировщик    • Контролер  │    │ • discovery по журналу         │
│ • Агент процесса (×N)         │    │ • conformance vs нормативная   │
│ • Шаблонизатор   • Интерфейсы │    │ • расхождения → корректировки  │
└───────────────────────────────┘    └────────────────────────────────┘
        │
Слой данных (PostgreSQL: модели, состояние, event_log, версии параметров;
             Redis Streams — шина; файловое хранилище документов)
```

Принципы: **детерминированное ядро** (никаких LLM в оркестровке и документах), **прослеживаемость** (каждое действие агента → сообщение → правило), **стандарты** (BPMN 2.0, FIPA ACL, XES), **воспроизводимость** (Docker Compose, фиксируемый seed), **только обезличенные/синтетические данные**.

## Быстрый старт (планируемый)

```bash
git clone https://github.com/<user>/adaptive-bpm-loop && cd adaptive-bpm-loop
docker compose up            # PostgreSQL + Redis + API + агенты + фронтенд
python -m simgen --cycles 8 --seed 42   # имитация 8 семестров
python -m mining.report                 # отчёт: метрики по циклам, до/после адаптации
```

## Ограничения

- Реальных журналов исполнения у вуза нет — эксперимент проводится на **синтетических журналах**, параметры которых калибруются по реальным нормативным документам кафедры и (по мере получения) фактическим датам одного выпуска; это зафиксировано как ограничение исследования.
- Классического машинного обучения в ядре нет — намеренно: интеллектуальная составляющая — методы анализа процессов (discovery/conformance). Предиктивный мониторинг сроков — возможное расширение.
- Персональные данные не используются: только обезличенные роли и синтетические исполнители.
