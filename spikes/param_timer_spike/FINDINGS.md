# Спайк параметризации таймеров через данные процесса (D1/T-50)

**Результат прогона:** 5/5 (`run_spike.py`)

## Вопрос

Контур адаптации (T-41/T-42) меняет нормативные сроки между циклами. Чтобы
это было измеримо в эксперименте (T-51), длительность граничных таймеров
BPMN должна браться из параметра процесса, переданного при запуске экземпляра
(`Orchestrator.start_instance(..., initial_data=...)`), а не быть константой
в самом BPMN-файле — иначе меняющиеся между циклами корректировки означали бы
перегенерацию BPMN-файла на каждый цикл.

## Вердикт: ✅ работает, но НЕ через `wf.data`

`timeDuration` в BPMN — Python-выражение (уже отмечено в спайке T-28: строка
пишется без внешних кавычек как выражение, например `f'P{reminder_days}D'`),
вычисляется `PythonScriptEngine.evaluate(task, expression)` в пространстве
имён **`task.data`**, не `workflow.data`.

## Находка: `wf.data` и `task.data` — независимые хранилища

`Task.__init__` (SpiffWorkflow/task.py:84) безусловно ставит `self.data = {}`
для ЛЮБОЙ задачи, включая корневую — никакого чтения из `workflow.data` при
создании нет. `workflow.data` синхронизируется с `task.data` только один раз
и только в одну сторону — при завершении процесса (`self.data.update(task.data)`
в `on_complete`, см. находку спайка T-28/п.3 в FINDINGS.md оркестратора).
То есть `wf.data.update(...)`, вызванный после создания `BpmnWorkflow(spec)`
(а корневая задача создаётся уже внутри конструктора — `Workflow.__init__`
строка 65), не может повлиять на исполнение вообще: дерево задач его просто
не видит.

**Правильный способ задать начальные данные экземпляра:**
`wf.task_tree.data.update(initial_data)` — дочерние задачи наследуют данные
родителя через `deepcopy` при создании (`task.py:175`), поэтому значения,
записанные в данные корневой задачи ДО `do_engine_steps()`, корректно
пропагируются вниз по всему дереву, включая задачи, создаваемые уже после
первого шага.

## Исправлен реальный баг в оркестраторе

`orchestrator/engine.py Orchestrator.start_instance(..., initial_data=...)`
использовал `wf.data.update(initial_data)` — параметр существовал в сигнатуре
с самого начала (B1, T-31), но НИКОГДА не оказывал влияния на исполнение,
потому что ни один прежний тест не передавал непустой `initial_data`, а если
бы передавал — значения тихо терялись бы. Исправлено на `wf.task_tree.data`.

## Применение для эксперимента (T-51)

Демо-модели процессов для эксперимента должны задавать длительности
контрольных точек как Python-выражения от переменных процесса (например,
`f'P{reminder_days}D'` / `f'P{escalation_days}D'`), а не константами `"P7D"`.
Тогда контур адаптации между циклами эксперимента просто меняет значения,
передаваемые в `initial_data` при следующем `start_instance`, без
регенерации BPMN-файлов.
