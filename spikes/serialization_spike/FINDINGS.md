# Спайк сериализации BpmnWorkflow (T-31, часть B1)

**Результат прогона:** 10/10 (`run_spike.py`)

## Вопрос

Переживает ли экземпляр процесса с активными таймерами (напоминание, эскалация)
сериализацию → запись в PostgreSQL → полное уничтожение объекта в памяти →
чтение из PostgreSQL → десериализацию — без потери таймеров и данных?

## Вердикт: ✅ да, полностью

`BpmnWorkflowSerializer.serialize_json()` / `.deserialize_json()` (SpiffWorkflow 3.1.2)
корректно сохраняют и восстанавливают состояние, включая ещё не сработавшие граничные
таймерные события. Таймеры на восстановленном объекте срабатывают по исходному графику
(отсчёт от момента старта задачи, а не от момента восстановления) — то есть время
хранится абсолютным, не относительным «сколько осталось».

## Решения для оркестратора (T-31)

1. **Хранить как JSONB**, не TEXT: `serialize_json()` отдаёт валидный JSON-текст,
   psycopg принимает его напрямую в jsonb-колонку.
2. **Persist-точки**: сохранять состояние после каждого `do_engine_steps()` —
   после старта, после каждого выполненного шага, после каждого сработавшего
   таймера. Не после `refresh_waiting_tasks()` самого по себе — там могут не быть
   изменений.
3. **Тик оркестратора** обязателен (найдено ещё в спайке T-28): без периодического
   `refresh_waiting_tasks()` + `do_engine_steps()` таймеры не проверяются. Для
   прототипа — фоновая задача с интервалом (например, раз в минуту); интервал
   определяет точность срабатывания контрольных точек.
4. Формат не привязан к версии кода спецификации BPMN «жёстко фатально» —
   `BpmnWorkflowSerializer(version=...)` поддерживает миграции (`.migrate()`),
   пригодится при изменении BPMN-моделей между циклами (корректировки контура
   адаптации меняют параметры, не саму модель, но на будущее стоит знать).
