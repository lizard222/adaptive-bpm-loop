# Спайк управляемого времени для симуляции (B5/T-35)

**Результат прогона:** 3/3 (`run_spike.py`)

## Вопрос

Нормативные сроки процессов кафедры измеряются днями/неделями («приказ о
допуске — за 14 дней до ГЭК»). SpiffWorkflow оценивает граничные таймеры через
`datetime.now(timezone.utc)` напрямую (`bpmn/specs/event_definitions/timer.py`),
без какой-либо инъекции часов. Можно ли прогнать восемь виртуальных семестров
без восьми семестров реального ожидания?

## Вердикт: ✅ да, через freezegun

`freeze_time(...)` патчит класс `datetime.datetime` глобально — в том числе
внутри уже импортированных модулей вроде `SpiffWorkflow.bpmn.specs.event_definitions.timer`,
которые сделали `from datetime import datetime, timezone` до того, как спайк
вообще запустился. Внутри `with freeze_time(start) as frozen:` вызов
`frozen.move_to(...)` мгновенно переносит часы на 15 дней вперёд, и таймер
`P14D` резолвится обычным `refresh_waiting_tasks() + do_engine_steps()` —
без единой секунды реального ожидания.

## Решения для генератора (simgen/, T-35)

1. **Критическая находка по ходу реализации, не по самому спайку**: схема
   `event_log` изначально использовала `ts TIMESTAMPTZ DEFAULT now()` —
   время **сервера PostgreSQL**, которое freezegun не патчит (это отдельный
   процесс). Под управляемыми часами это слепило бы события всех виртуальных
   дней в одну реальную метку времени — журнал стал бы бесполезен для анализа.
   Исправлено: `orchestrator/engine.py._log_event` теперь пишет `ts` явно из
   Python (`datetime.now(timezone.utc)`, которое freezegun ловит), передавая
   его в INSERT вместо того чтобы полагаться на DEFAULT. Для обычной (не
   симулированной) работы оркестратора/агентов поведение не изменилось.
2. **Шаг симуляции — виртуальные сутки**, не точный момент следующего
   срабатывания таймера: для сроков в днях/неделях этого достаточно, и не
   нужно лезть во внутренние данные SpiffWorkflow, чтобы вычислить точное
   время due у конкретного boundary-таймера.
3. Готовые нормативные модели кафедры (T-22/T-23) можно будет писать сроками
   `P7D`/`P14D` и т. п. без опасений за скорость симуляции — то же самое
   `"PT2S"` в спайке T-28 было временной заменой для тестов на секундах,
   реальным моделям она была не нужна с самого начала.
