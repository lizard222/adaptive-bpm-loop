# Спайк SPADE + Prosody (B3/B4, инфраструктура для FIPA ACL)

**Результат прогона:** 6/6 (`run_spike.py`)

## Вопрос

Поднимается ли реальная связь между двумя SPADE-агентами через локальный XMPP-сервер
(Prosody в docker-compose), с доставкой FIPA-метаданных (performative)?

## Вердикт: ✅ да, но потребовалось три последовательных фикса

Ни один из них не был очевиден из документации SPADE — каждый нашёлся через логи
Prosody и чтение исходников slixmpp.

### 1. Образ `prosody/prosody:latest` не может прочитать собственный TLS-сертификат

`localhost.key` в образе принадлежит `root` с правами `rw-------`; процесс Prosody
работает под пользователем `prosody` (см. `entrypoint.sh`: `setpriv --reuid=prosody`)
и не может его открыть. Это баг/особенность образа, не нашей конфигурации.
**Решение:** не чинить сертификат — модуль `"tls"` просто не включён в
`modules_enabled` (`infra/prosody/prosody.cfg.lua`). Для локального прототипа в
доверенной сети (docker-compose/localhost) это приемлемо; для чего-то похожего на
продакшен нужен нормальный сертификат.

### 2. Без TLS сервер по умолчанию не предлагает SASL PLAIN

Добавлено `allow_unencrypted_plain_auth = true` в конфиг Prosody. Само по себе не
хватило (см. следующий пункт), но необходимо.

### 3. Клиент (slixmpp) отдельно, вне зависимости от сервера, отказывается слать
PLAIN по незашифрованному каналу

Это не настройка сервера, а политика самого клиента: плагин `feature_mechanisms`
внутри slixmpp имеет `default_config['unencrypted_plain'] = False` и на этом
основании кидает `SASLCancelled('PLAIN without encryption')` — с сообщением
`"No appropriate login method"` на стороне SPADE. SPADE не даёт публичного способа
это переопределить (клиент `XMPPClient` создаётся заново внутри `Agent.start()`,
без hook-точки между созданием и подключением).

**Решение:** `agents/xmpp_dev.py` — подменяет класс `spade.agent.XMPPClient` на
подкласс, который сразу после `super().__init__()` выставляет
`self["feature_mechanisms"].unencrypted_plain = True`. Вызывается один раз в начале
процесса: `enable_unencrypted_plain_auth()`, до создания любых `Agent(...)`.

## Решения для агентов (T-33, T-34) и дальше

1. Любой скрипт/процесс, создающий SPADE-агентов для этого проекта, должен в самом
   начале импортировать и вызвать `agents.xmpp_dev.enable_unencrypted_plain_auth()`
   — до `from spade.agent import Agent` фактически неважно, но до **создания**
   первого `Agent(...)` обязательно.
2. `auto_register=True` (значение по умолчанию) работает: агент сам создаёт учётку
   на Prosody при первом подключении — отдельно предрегистрировать JID не нужно.
3. Формат JID — `<имя>@localhost`, порт по умолчанию 5222 (проброшен в
   docker-compose и на хост, и подходит агентам, запущенным вне контейнеров).
4. FIPA-подобные метаданные (`performative`, `protocol`, ...) хранятся как
   `Message.set_metadata(key, value)` — обычные строки, без специальной XEP-обвязки;
   этого достаточно для целей NFR-3 (используется терминология FIPA ACL).
5. Если это когда-нибудь потребуется вынести за пределы локальной dev-сети —
   решение №1 (без TLS) придётся пересмотреть: сначала почини сертификат Prosody
   (смонтировать свой валидный self-signed или полноценный), затем убрать
   `agents/xmpp_dev.py` и `allow_unencrypted_plain_auth`.
